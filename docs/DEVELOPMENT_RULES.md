# 開発ルール (Development Rules)

このドキュメントは、AIコーディングアシスタント（Claude Code, Gemini/Antigravity, Cursor等）と人間が協力して開発を行う際の共通ルールを定めたものです。

---

## 🧠 AImemory 参照（最重要）

> **作業開始時に必ず `C:\Tools\AImemory\index.md` を読んでください。**
>
> このファイルには過去のプロジェクトで蓄積されたバグ・失敗事例・ベストプラクティスが記録されています。
> 現在の作業に関連するエントリがあれば、`C:\Tools\AImemory\entries\` ディレクトリ内の該当ファイルも必ず読んでください。
>
> 詳細な利用手順: `C:\Tools\AImemory\AI_INSTRUCTIONS.md`

---

## 1. 基本原則

### 1.1 作業の進め方

1. **計画を先に立てる** — いきなりコードを書かず、まず実装計画を提示してユーザーの承認を得る
2. **段階的に実装する** — 大きな変更は小さなステップに分割し、各ステップで動作確認する
3. **既存コードを尊重する** — 既存のコードスタイル・パターンに合わせる。独断で大幅なリファクタリングをしない
4. **確認してから破壊する** — ファイル削除・大規模変更の前に必ずユーザーに確認する

### 1.2 コミュニケーション

- 不明点は推測せず質問する
- 複数の実装方針がある場合は選択肢を提示する
- エラーが発生した場合は原因と対処法を説明する
- 作業完了時は変更内容のサマリーを報告する

## 2. コーディング規約

### 2.1 全般

- **言語**: プロジェクトで使用される言語に従う。コメントは日本語OK
- **インデント**: スペース（幅はプロジェクトの既存コードに合わせる。新規の場合は2 or 4スペース）
- **文字コード**: UTF-8（BOMなし）
- **改行コード**: LF（Windows環境でもLFを推奨。.gitattributesで制御）
- **末尾改行**: ファイル末尾には空行を1つ入れる

### 2.2 命名規則

| 対象 | 規則 | 例 |
|------|------|-----|
| ファイル名 | ケバブケース or スネークケース | `user-service.ts`, `user_service.py` |
| クラス名 | パスカルケース | `UserService` |
| 関数/メソッド | キャメルケース or スネークケース | `getUserById`, `get_user_by_id` |
| 定数 | アッパースネークケース | `MAX_RETRY_COUNT` |
| 変数 | キャメルケース or スネークケース | `userName`, `user_name` |
| CSS クラス | ケバブケース or BEM | `card-header`, `card__title--active` |

### 2.3 コメント

- **TODOコメント**: `// TODO: 説明` の形式で統一
- **FIXMEコメント**: `// FIXME: 説明` の形式で統一
- **マジックナンバー禁止**: 意味のある定数名を付ける
- **自明なコメントは不要**: コードで自明なことをコメントしない
- **Whyを書く**: 「何をしているか」ではなく「なぜそうしているか」を書く

---

## 3. ファイル・ディレクトリ構成

### 3.1 基本ルール

- 1ファイルに1つの責務（単一責任原則）
- ファイルは300行以内を目安にする（超える場合は分割を検討）
- テストファイルは対象ファイルと同じディレクトリ構造で `__tests__/` or `tests/` に配置
- 設定ファイルはプロジェクトルートに配置

### 3.2 よく使うディレクトリ構造

```
project/
├── src/                 # ソースコード
│   ├── components/      # UIコンポーネント
│   ├── services/        # ビジネスロジック
│   ├── utils/           # ユーティリティ関数
│   ├── types/           # 型定義
│   └── constants/       # 定数定義
├── tests/               # テストコード
├── docs/                # ドキュメント
├── scripts/             # ビルド・デプロイスクリプト
└── config/              # 設定ファイル
```

## 4. Git 運用

### 4.1 コミットメッセージ

[Conventional Commits](https://www.conventionalcommits.org/) に従う：

```
<type>(<scope>): <summary>

<body>（任意）
```

**type 一覧:**

| type | 用途 |
|------|------|
| `feat` | 新機能 |
| `fix` | バグ修正 |
| `docs` | ドキュメント変更 |
| `style` | フォーマット変更（動作に影響なし） |
| `refactor` | リファクタリング |
| `test` | テスト追加・修正 |
| `chore` | ビルド・設定変更 |

### 4.2 ブランチ運用

- `main` — 安定版。直接pushしない
- `develop` — 開発ブランチ
- `feature/<name>` — 機能ブランチ
- `fix/<name>` — バグ修正ブランチ

---

## 5. エラーハンドリング

- **例外を握りつぶさない** — catch したらログ出力 or 再スロー
- **ユーザー向けエラーと開発者向けエラーを分離する**
- **早期リターン (Guard Clause)** を活用して深いネストを避ける
- **エラーメッセージは具体的に** — 「エラーが発生しました」ではなく、何が原因で何をすべきかを含める

---

## 6. テスト方針

- **新機能にはテストを書く** — ユニットテストを基本とする
- **バグ修正時は再発防止テストを書く**
- **テストは独立して実行可能にする** — テスト間の依存を作らない
- **テスト名は何をテストしているか分かるように** — `test_ユーザー作成時にメールが送信される`

---

## 7. セキュリティ

- **秘密情報をコードにハードコードしない** — 環境変数 or `.env` ファイル（`.gitignore` に追加）
- **ユーザー入力は必ずバリデーションする**
- **依存パッケージは定期的にアップデートする**
- **`.env`ファイルは `.gitignore` に含める**

---

## 8. パフォーマンス

- **早すぎる最適化はしない** — まず正しく動かし、必要に応じて最適化
- **N+1問題に注意** — DBクエリの発行回数を意識する
- **大量データの処理** — ページネーション・ストリーミングを検討

---

## 9. AI特有の注意事項

### 9.1 AIが避けるべきこと

- ユーザーの確認なしに依存パッケージのメジャーバージョンを変更しない
- 既存のテストを無断で削除・スキップしない
- `.env` / 秘密鍵 / 認証情報を出力しない
- 動作確認せずに「完了」と報告しない

### 9.2 AIが積極的にすべきこと

- 変更による影響範囲を報告する
- 型安全性を意識する（TypeScript, Python type hints等）
- エッジケースを考慮する
- 既存のテストが通ることを確認する
- AImemoryに記録すべき知見を見つけたら提案する

---

## 10. Windows環境固有の注意事項

- **パス区切り**: `\` と `/` の混在に注意。可能な限り `path.join()` 等を使う
- **改行コード**: `.gitattributes` で `* text=auto eol=lf` を設定
- **文字コード**: バッチファイルの文字コードに注意（`chcp 65001` で UTF-8 に）
- **ファイル名**: 大文字小文字を区別しないことに注意
- **長いパス**: `\\?\` プレフィックスや LongPathsEnabled を検討
